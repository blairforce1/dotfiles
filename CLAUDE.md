# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

```sh
./bootstrap.sh   # one-time fresh-machine setup (installs Nix, first switch)
./init.sh        # generates local.nix (OS username + git identity), gitignored
./rebuild.sh     # apply changes to home.nix / flake.nix (day-2 use)
./update.sh      # bump pinned flake inputs to latest (add input name(s) to update only those); follow with ./rebuild.sh to apply
nix flake check path:.  # validate without applying -- needs local.nix; path: not bare .
nix build "path:$(pwd)#homeConfigurations.$(whoami).activationPackage" --dry-run  # dry-run build
```

Every Nix command against this flake must use a `path:` ref
(`path:.`/`path:$(pwd)`), never the bare `.`/directory form — see
Architecture and Deliberate decisions below for why.

Editing a file under `home/` takes effect immediately — those paths are real
symlinks (`mkOutOfStoreSymlink`) into `$HOME`, not copies. `rebuild.sh` is
only needed after changing `home.nix` or `flake.nix` themselves.

Before adding any package to `home.nix`, confirm it actually exists in the
pinned revision — `nix search nixpkgs <name>` queries the flake registry's
default nixpkgs, which is **not** the same commit this repo pins, and a false
positive there wastes a full failed `rebuild.sh` (`error: undefined variable`).
Check the real pinned revision instead:

```sh
REV=$(python3 -c "import json; print(json.load(open('flake.lock'))['nodes']['nixpkgs']['locked']['rev'])")
nix eval "github:NixOS/nixpkgs/$REV#legacyPackages.x86_64-linux.<name>.version"
```

Swap `nodes['nixpkgs']` for `nodes['nixpkgs-unstable']` to check the other
pinned input. If a package is missing from stable but present in unstable
(common for fast-moving tools — this is exactly why `herdr` and `vscode` are
sourced from `pkgsUnstable`), that's the fix; don't bump the stable pin just
to get one new package.

## Architecture

This repo is a standalone home-manager profile driven by a Nix flake. It
deliberately manages only the user's home directory, not the OS — there is
no NixOS or nix-darwin module here, and Fedora/dnf/KDE stay authoritative for
system-level config.

- **`flake.nix`** pins inputs and defines `homeConfigurations.<user>`, where
  `<user>` and the git identity both come from `local.nix` (gitignored,
  generated by `./init.sh` — see Deliberate decisions). No fallback: a
  missing `local.nix` throws a clear error pointing at `./init.sh`. It also
  builds two separate `pkgs` instances: the stable one used for almost
  everything, and `pkgsUnstable` for packages that move faster than the
  stable channel backports (`vscode`; `herdr`, a weekly-release tool not yet
  in the stable snapshot). Both are built directly with
  `config.allowUnfree = true` rather than through home-manager's
  `nixpkgs.config`, which is a no-op once a pre-built `pkgs` is passed in —
  needed for the proprietary Pylance extension.
- **`home.nix`** is the actual profile: packages, shell (bash + oh-my-posh +
  fzf + zoxide + direnv), git (identity from `local.nix` via `flake.nix`,
  see below), and a fully declarative VS Code (settings + extensions, sourced from the
  `nix-vscode-extensions` marketplace mirror rather than nixpkgs' own curated
  set). It's kept as a single file by design — see Deliberate decisions.
- **`home/`** holds the actual dotfile *contents*. `home.nix` only wires each
  one into `$HOME` via `home.file.<path>.source = mkOutOfStoreSymlink
  "${dotfiles}/home/<path>"`, where `dotfiles = ~/.dotfiles`. To add a new
  managed dotfile: add the file under `home/`, add one `home.file` line in
  `home.nix`, run `rebuild.sh`.
  - `home/.claude/CLAUDE.md` and `home/.claude/settings.json` are symlinked
    to `~/.claude/` — this repo manages the user's *global* Claude Code
    config. That's a separate artifact from this file, which only applies
    to work inside this repo.
  - `home/.config/{wezterm,herdr,lazygit}/...` are real, working configs —
    see the recipe below for the pattern they follow.
  - `home/.config/starter/config` is the original placeholder that proved
    the symlink mechanism before any real dotfile existed; harmless to
    replace or remove, not load-bearing.
- **`~/.dotfiles`** is a fixed symlink to this repo's clone location,
  maintained by both `bootstrap.sh` and `rebuild.sh`. The `mkOutOfStoreSymlink`
  calls in `home.nix` resolve through this fixed path, not wherever the repo
  actually lives, so it must exist before any switch.
- **`init.sh`**: generates `local.nix` (gitignored) with the OS username and
  git identity this flake requires. Auto-detects where it can (existing
  global git config, authenticated `gh` user, or the repo's own `origin`
  remote owner), always shows the detected value for confirmation/override
  rather than applying it silently, and prompts for whatever it can't
  detect. Re-runnable to change identity later.
- **`bootstrap.sh`** (fresh machine, one-time): installs Determinate Nix,
  creates `~/.dotfiles`, runs `./init.sh` if `local.nix` doesn't exist yet,
  then runs the first `home-manager switch` via `nix run` (home-manager
  isn't on PATH yet at this point) against a `path:` flake ref.
- **`rebuild.sh`** (day-2): re-links `~/.dotfiles`, fails fast with a clear
  message if `local.nix` is missing, and runs `home-manager switch -b
  backup --flake "path:$DIR#$(whoami)"` directly.

## Adding a new managed CLI tool

The pattern used for WezTerm, herdr, and lazygit — follow it rather than
reinventing a per-tool approach:

1. Check the package exists in the pinned nixpkgs revision (see Commands
   above). Use `pkgsUnstable.<name>` instead of plain `<name>` if it's only
   in the unstable input.
2. Check whether home-manager has a `programs.<tool>` module (search or
   check `home-manager/modules/programs/<tool>.nix` at the pinned rev:
   `https://raw.githubusercontent.com/nix-community/home-manager/<rev>/modules/programs/<tool>.nix`,
   `rev` from `flake.lock`'s `home-manager` node — more reliable than
   trusting a search summary). If one exists, use it **only** for
   `enable`/`package`/`enableBashIntegration`-style options — never its
   `settings`/`extraConfig`, which would write the real config into the Nix
   store as a generated file, losing syntax highlighting and requiring
   `rebuild.sh` for every tweak.
3. The actual config content is a real file under `home/.config/<tool>/...`,
   wired into `$HOME` via one more `home.file` + `mkOutOfStoreSymlink` line
   in `home.nix` (same as every other entry there).
4. For the config's exact schema/syntax, once the tool is installed, prefer
   its own introspection (`--default-config`, `--print-config`, `--help`,
   etc.) over web docs — it's authoritative for the exact installed version
   and avoids hallucinated option names. `herdr --default-config` and
   `lazygit --config` are both fully-commented references this way.
5. Verify end-to-end before considering it done: `nix flake check path:.` →
   `nix build "path:$(pwd)#homeConfigurations.$(whoami).activationPackage"
   --dry-run` → `rebuild.sh` → confirm the binary resolves (`command -v
   <tool>`) → confirm the symlink resolves back into this repo (`readlink -f
   ~/.config/<tool>/...`) → sanity-parse the config file in its actual
   format.

## Gotchas

- **When moving a `home.file` entry to a different path granularity (e.g.
  directory-level symlink → file-level symlink for the same location),
  remove the stale `$HOME`-side symlink before the next `rebuild.sh`.**
  home-manager's collision check follows existing symlinks to see if the
  target already matches, and if the old link chain still resolves into
  this repo, `rebuild.sh` can write *through* it and silently corrupt the
  real source file in `home/` (it happened once, while developing the Vale
  config below — home-manager "backed up" the actual repo file to
  `<file>.backup` and replaced it with a symlink pointing back into a Nix
  store path that resolved back to itself, an actual symlink cycle). Always
  `rm` the old `$HOME` path first, then `rebuild.sh`, and check
  `readlink -f`/`file` on the result before trusting it.
- **`warning: ...builtins.derivation... 'options.json'... without a proper
  context` during `rebuild.sh` is benign.** Previously (before this repo
  switched every Nix invocation to `path:` refs) this only appeared with
  uncommitted changes in the tree, since `git+file:` sources are
  content-addressed by commit and a dirty tree has no clean commit to
  address. `path:` sources were never content-addressed that way regardless
  of git state, so this warning may now show up unconditionally, dirty tree
  or not — still benign either way, activation completes correctly.
- **Shell-integration changes need a brand-new shell, not a reused one.**
  After `rebuild.sh` changes anything shell-related (keybindings, prompt
  hooks, completions), an already-open shell keeps its old bindings/hooks
  loaded — symptoms look like the change didn't take effect (e.g. Ctrl+R
  still offering to install a tool that was just removed) when it actually
  did. `rebuild.sh` automates the fix when run inside WezTerm (opens a fresh
  tab via `wezterm cli spawn` and closes the old one via `kill-pane`,
  guarded on `$WEZTERM_PANE`); in any other terminal, open a new tab/window
  manually after it finishes.
- **`~/.gitconfig` and `~/.gitconfig.local` must not exist.** They're
  pre-Nix-migration leftovers (removed to `~/.gitconfig.pre-nix.bak` /
  `~/.gitconfig.local.pre-nix.bak`, not in this repo) that used to shadow
  the Nix-managed `~/.config/git/config` — git reads `~/.gitconfig` *after*
  the XDG path and it wins on conflicts. While they existed, every commit
  landed under the OS account's real-name GECOS fallback instead of the
  configured git identity (see below), and a broken
  `credential.helper = "!gh auth token"` in the old
  file caused a mystery `accepts 0 arg(s), received 1` after every push. If
  either file ever reappears (e.g. `gh auth setup-git` or similar recreates
  it), delete it rather than editing it — the real config lives in
  `home.nix`.

## Deliberate decisions

Notes for coding agents — these are intentional choices, not oversights.
Don't "helpfully" revert them.

- **This repo carries no identity of its own — OS username and git
  `user.name`/`user.email` both come from `local.nix`, gitignored and
  generated by `./init.sh`.** `flake.nix` throws a clear error if
  `local.nix` is missing; there is no fallback value anywhere. Two earlier
  approaches were tried and rejected: (1) a hardcoded default identity
  baked into `flake.nix`/`home.nix` — worked, but meant sharing this repo
  meant sharing personal identity strings in tracked files; (2)
  `builtins.getEnv`-sourced env vars (`$USER`, `GIT_USER_NAME`,
  `GIT_USER_EMAIL`) with `~/.dotfiles.env` sourced by both scripts — no
  hardcoded identity, but needed `--impure` on every single Nix invocation
  forever, plus export/source plumbing duplicated in both scripts. Landed
  on `local.nix` because it needs neither: `import ./local.nix` is a normal
  pure Nix operation, and reading it requires only that every Nix
  invocation use a `path:` flake ref (`path:.` / `path:$DIR`) instead of
  the bare `.`/directory form — `path:` copies the whole directory
  verbatim, while the default `git+file:` a bare `.` resolves to (for a
  git-repo directory) only sees git-tracked files, which is why `local.nix`
  being gitignored requires `path:` specifically. Confirmed empirically with
  a throwaway gitignored probe file and a temporary flake output checking
  `builtins.pathExists`: `false` under a bare `.` ref (git+file), `true`
  under `path:$(pwd)`, no `--impure` needed either way.
- **A `.githooks/pre-commit` hook (wired via `git config core.hooksPath
  .githooks` in `init.sh`) blocks commits containing a literal `/home/<user>`
  path in tracked files.** Added after the pattern leaked into
  `settings.json` twice (a hook command and a statusLine command both
  hardcoded the literal home path instead of expanding `$HOME` at runtime). Fix on
  trigger: use `$HOME` in shell command strings or
  `config.home.homeDirectory` in Nix, not a literal path. Deliberate,
  non-identifying example path in a doc/comment can bypass with `--no-verify`.
- **`init.sh` auto-detects git identity where it safely can, but always
  shows the detected value for confirmation rather than applying it
  silently.** Cascade: existing global git config, then the authenticated
  `gh` CLI user, then the `origin` remote's owner segment (works once
  someone's forked+cloned; yields a handle, not a real name). Deliberately
  does *not* fall back to git/passwd's own real-name auto-detection — see
  the `~/.gitconfig` gotcha below for exactly what that caused last time.
- **No system-level Nix module.** Fedora/dnf/KDE stay authoritative for
  system config on purpose. Don't add a nix-darwin or NixOS module unless
  that's an explicit, separate decision.
- **`home.nix` is a single file.** Not split into `home/modules/*.nix`. Keep
  it that way until it's actually too big to read comfortably in one
  sitting — don't restructure preemptively.
- **`targets.genericLinux.enable = true;` is required**, not optional
  cruft — Fedora is not NixOS, so home-manager needs this to wire up things
  like `XDG_DATA_DIRS` correctly.
- **`home.stateVersion` is pinned once at first install.** It is not a
  "current version" field and should not be bumped just to match whatever
  release is newest.
- **fzf, not atuin, for shell history.** Deliberately rejected atuin after
  trying it — disliked its smart-ranking sort order and the intrusive
  full-screen search UI. Don't re-propose it without being asked.
- **Git credential helpers must speak the credential-helper protocol.** Git
  appends `get`/`store`/`erase` as a positional arg when invoking a
  configured `credential.helper`. `gh auth git-credential` is designed for
  this; a plain `gh auth token` is not (it takes no positional args at all)
  and errors if used as a helper. Don't add `credential.helper = "!gh auth
  token"` — a previous version of this file had exactly that, and it's what
  caused the `~/.gitconfig` gotcha above.
